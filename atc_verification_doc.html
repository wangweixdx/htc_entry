Part 1: 验证环境设计

设计概述

基于 SystemVerilog + UVM-lite 框架的模块级验证环境，采用不带 UVM 库的纯 SV 实现（兼容所有主流仿真器）。环境由 3 个 Agent（DMA/ATS/CSR）、1 个 Scoreboard、1 个 Monitor 和若干 Checker 组成。

1.3 Scoreboard 参考模型设计

参考模型核心逻辑

Scoreboard 内部维护一个影子 ATC（2048 entry 的软件模型），实时镜像 DUT 的 TAG+SRAM 状态：

  1. LOOKUP 时：参考模型对输入地址执行与 DUT 相同的 hash+compare 逻辑，预测 hit/miss + 翻译地址 + 64K pre-hit，与 DUT 输出比对
  2. INSERT 时：参考模型更新影子 entry（含 dupcheck 后覆盖或 NRU 分配），比对 DUT 的 dupcheck 结果
  3. INVALIDATE 时：参考模型清除匹配的 entry，比对 DUT 的 inv_ack/inv_hit_idx
  4. FLR/ATS Toggle 时：参考模型批量清除，比对 DUT 完成信号
Scoreboard 影子 ATC 数据结构 (per entry):
┌────────┬──────┬────────┬────────┬──────────┬─────┬─────────────────┬──────┐
│ valid  │  PV  │ PASID  │ FuncID │   GVA    │ STU │ translated_addr │ perm │
│  1b    │ 16b  │  16b   │  16b   │   64b    │ 5b  │      64b        │  4b  │
└────────┴──────┴────────┴────────┴──────────┴─────┴─────────────────┴──────┘

1.4 Checker 设计 (SVA 断言)

断言名类型描述
CHK_PIPELINE_LATENCY时序查表请求 → 响应严格 3 cycle latency
CHK_LOOKUP_THROUGHPUT吞吐连续请求时每个 cycle 产生一个响应 (pipeline fill 后)
CHK_HIT_CONSISTENCY一致性命中时 translated_addr 必须等于最近一次 insert 的值
CHK_INV_ACK_TIMING时序invalidation 完成后必须在规定 cycle 内返回 ACK
CHK_FLR_CLEANUP功能FLR 后对应 func_id 的所有 entry 不再命中
CHK_ATS_TOGGLE_CLEAN功能ATS disable 后所有查表返回 miss
CHK_DUPCHECK_RESULT功能重复地址 insert 时 dupcheck=1；新地址时 dupcheck=0
CHK_NRU_EVICTION功能set 满时 insert 正确替换 NRU victim；被替换的 entry 不再命中
CHK_64K_PREHIT功能当 addr+64K 已缓存时，pre_hit 正确置位
CHK_PV_ISOLATION安全PV 不匹配时绝不命中；不同 VF 间无地址泄漏

Part 2: 验证计划

验证策略

采用 定向测试 + 约束随机测试 混合策略。先用定向测试覆盖基础功能和边界条件，再用约束随机测试提升覆盖率。覆盖率驱动验证 (CDV) 闭环。

2.4 验证流程

Phase 1: 冒烟 (Day 1-2)
  ├── TC_SMK_01 ~ 04
  └── 目标：DUT 基本连通，无编译/elab 错误

Phase 2: 定向功能 (Day 3-7)
  ├── TC_LU_01 ~ 13 (查表全功能)
  ├── TC_INS_01 ~ 08 (插入全功能)
  ├── TC_INV_01 ~ 08 (失效全功能)
  ├── TC_ARB_01 ~ 05 (仲裁全功能)
  └── 目标：所有定向用例 PASS，功能覆盖 60%

Phase 3: 边界条件 (Day 8-10)
  ├── TC_CRN_01 ~ 10
  └── 目标：边界用例 PASS，无死锁/竞争

Phase 4: 随机回归 (Day 11-14)
  ├── TC_RND_01 ~ 05 (大量随机 seed)
  └── 目标：功能覆盖 100%，代码覆盖达标

Phase 5: 收尾 (Day 15)
  ├── 覆盖率 gap 分析
  ├── 定向补充用例
  └── 目标：全部覆盖率达标，签收
