ATC (Address Translation Cache) 模块设计文档

EP 侧 · PCIe Gen6 · 2048 Entry · 64-way Set-Associative · SRIOV · PV=1 PASID=20 · 17-addr Prefetch · SF4X @1GHz · v2.4

1. 模块概述

项目定位

本模块是 Endpoint (EP) 侧 的 ATC (Address Translation Cache) 控制器，部署于 PCIe Endpoint 设备中。它缓存来自 Root Complex (RC) 中 Translation Agent (TA) 的地址转换结果，为 EP 内部 DMA 引擎提供高速地址翻译查询服务。

核心功能
  1. 地址转换缓存：接收来自 RC 的 translation completion，将 {PV, PASID, Function ID, 未转换地址 → 已转换地址} 映射存入 entry
  2. 并行查表：对 DMA 引擎发来的 lookup 请求进行 hash → set 选择 → way 并行比较 → 命中判断，返回 hit/miss + 翻译结果
  3. 17-地址并行比较：每次 lookup 并发生成 17 个地址 (当前 + 4KB~64KB 共16个预取偏移)，64 way × 17 addr = 1088 路并行比较，提前发现连续地址命中
  4. 地址重复检测：新地址写入前遍历检查是否已存在，避免重复分配 entry
  5. 地址失效处理：响应 RC 发来的 ATS Invalidation Request，清除匹配的 entry
  6. 17-地址预取 (v2.0)：每次 lookup 并行比较 17 个地址（当前 + 4KB~64KB 共16个预取），per-function 使能
  7. Flow-control Ready 信号 (v2.2 / v2.3 增强)：dma_lu_req_ready/dma_rl_req_ready/ats_inv_req_ready/ats_comp_ready，优先级门控 — 高优先级请求活跃时低优先级 ready 自动 deassert
  8. FLR Done 信号 (v2.2)：csr_flr_req_done 输出，FLR 操作完成指示
  9. Prefetch Valid 信号 (v2.2)：prefetch_rsp_valid[15:0] 输出，指示预取结果有效性
  10. Relook 接口 (v2.1)：DMA 第二次查找接口，仅返回当前地址命中结果（不含预取），优先级介于 Insert 和 Lookup 之间
  11. Per-function ATS Enable (v2.0)：每个 Function 独立 1-bit ats_enable，64-bit 向量输入
  12. 特殊场景失效：ATS enable 切换、FLR (Function Level Reset) 时批量清除对应 Function 的全部 entry

4.2 请求仲裁优先级

优先级请求类型来源说明
0 (最高)FLRPCIe FLR 控制Function Level Reset，必须立即清除，阻塞其他所有请求
1ATS ToggleCSR / Config SpaceATS enable 边沿变化，批量清除对应 Function entry
2InvalidationRC (ATS Inv Request)RC 要求失效特定 translation，需尽快响应 (ATS 规范有时延要求)
3InsertRC (ATS Completion)新 translation 写入缓存
4 (最低)LookupDMA EngineDMA 地址翻译查询，频率最高但优先级最低

4.3 执行流水线 (4-stage @1GHz) 【v2.3 更新】

Stage操作关键路径预估延迟 (SF4X)
S0: Dispatch请求仲裁 → hash 计算 → set 索引解码 → 选中目标 sethash XOR + partition_hash() + decoder (5-to-32)~200ps
S1: TAG CompareTag 寄存器读取 → 64-way × 17-addr 并行比较 → hit 向量生成 → SRAM rd_en 发出64b comparator × 64 way × 17 addr + hit encoder~600ps
S2: SRAM ReadSync SRAM 数据返回 (1-cycle 读延迟) → 数据寄存器捕获SRAM read latency (1 cycle)~200ps
S3: Result响应组装 → 输出到 DMA / ATS 接口Response mux + output reg~150ps

设计结论：4-stage 流水线，latency = 4 cycles (4ns @1GHz)，throughput = 1 lookup/cycle。SRAM 改为同步读 (rd_en 在 S1 发出，数据在 S2 返回)，S1 关键路径 (~600ps) 满足 1ns 周期约束。查找、重复检测、失效三条路径共用此流水线。

5. 功能详细描述

5.1 地址查找与命中判断

查表信息包括：PV, PASID, device_function_ID, 当前未转换地址 (+64K 地址)

比较规则 (与原始设计一致，已确认)：
┌─────────────────────────────────────────────────────────────────────┐
│  IF  查表PV ≠ 表中PV → 未命中（MISS）                              │
│                                                                     │
│  ELIF 查表PV = 表中PV AND PV有效 →                                  │
│       比较 PASID AND device_function_ID AND untranslated_addr      │
│       全部通过 → 命中（HIT）；任一不通过 → 未命中（MISS）          │
│                                                                     │
│  ELSE (查表PV = 表中PV AND PV无效) →                               │
│       比较 device_function_ID AND untranslated_addr                │
│       通过 → 命中（HIT）；不通过 → 未命中（MISS）                  │
└─────────────────────────────────────────────────────────────────────┘

地址比较时：根据 STU 对查表地址和存储地址进行掩码处理后再比较。
掩码生成：addr_masked = addr & ~((1 << (STU + 12)) - 1)
// STU=0 → 4KB 对齐 (mask 低 12 bit)；STU=1 → 8KB 对齐 (mask 低 13 bit)
// STU=N → 2^(N+12)-byte 对齐 (PCIe ATS 标准定义)

5.2 17-地址并行比较 + 预取机制 (v2.0 / v2.3 更新)

每次 lookup 生成 17 个比较地址 (当前 + 16 个预取偏移)：
  ea_cmp_addr[0]  = lu_req_va + 64'h0000_0000_0000_0000   // 当前地址
  ea_cmp_addr[1]  = lu_req_va + 64'h0000_0000_0000_1000   // +4KB
  ea_cmp_addr[2]  = lu_req_va + 64'h0000_0000_0000_2000   // +8KB
  ...
  ea_cmp_addr[15] = lu_req_va + 64'h0000_0000_0000_F000   // +60KB
  ea_cmp_addr[16] = lu_req_va + 64'h0000_0000_0001_0000   // +64KB (64K 预查)

每个 way 配备 17 套地址比较器 (generate 循环)：
  ┌──────────────────────────────────────────────────────────────┐
  │  Way[k] Tag ──┬── cmp(ea_cmp_addr[0],  entry_addr) ──► hit[0]  │
  │               ├── cmp(ea_cmp_addr[1],  entry_addr) ──► hit[1]  │
  │               ├── ...                                          │
  │               ├── cmp(ea_cmp_addr[16], entry_addr) ──► hit[16] │
  │               │                                                │
  │  同时比较 PV, PASID, FuncID (共享一套比较器)                    │
  └──────────────────────────────────────────────────────────────┘

命中向量编码：
  ea_hit_vectors[16:0][63:0] — 17 组 64-bit hit vectors
  ea_hit_vectors[0]  对应当前地址命中 (exact hit)
  ea_hit_vectors[16] 对应 64KB 预查命中 (pre_hit)
  ea_hit_vectors[1:15] 对应 16 个预取偏移命中 (prefetch_hit[i])

响应信号语义：
  dma_lu_rsp_hit        = |ea_hit_vectors[0]  (仅当前地址精确命中)
  dma_lu_rsp_pre_hit     = |ea_hit_vectors[16] && !|ea_hit_vectors[0]  (仅 64KB，不含当前)
  prefetch_hit[15:0]     = {|ea_hit_vectors[16], ..., |ea_hit_vectors[1]}  (16 偏移独立指示)
  dma_rl_rsp_hit        = |ea_hit_vectors[0]  (Relook: 仅当前地址)

参数化：PREFETCH_COUNT=17, LOOKAHEAD_OFFSET=64'h1_0000 (64KB)


5.2.1 prefetch_enable 门控与 Hash 切换 【v2.4】

当 csr_prefetch_enable[func_id]==1 时, 除输出门控外,
查表和插入路径的 hash 算法也会切换:

// prefetch_enable=0: hash by 4KB-aligned VA
set_idx = partition_hash(func_id[5:0], func_id, va)
        // partition_hash 内部: func_id[3:0] ^ va[15:12]

// prefetch_enable=1: hash by 64KB-aligned VA
hash_va = {va[63:20], va[19:16], va[19:16], va[11:0]}
set_idx = partition_hash(func_id[5:0], func_id, hash_va)
        // partition_hash 内部: func_id[3:0] ^ va[19:16]


使用 64KB 对齐 hash 可保证当前地址和全部 16 个预取地址 (current+4KB ~ current+60KB)
落入同一个 set, 使预取命中能被正确检测。prefetch_enable=0 时仍使用 4KB 对齐 hash。


prefetch 输出信号通过 prefetch_enable 门控:


prefetch_rsp_valid = lu_engine_rsp_prefetch_hit & {16{lu_engine_rsp_valid && rsp_prefetch_en}};
prefetch_hit        = lu_engine_rsp_prefetch_hit & {16{rsp_prefetch_en}};
// dma_lu_rsp_pre_hit 同样被 prefetch_enable 门控

5.3 地址重复检测 (Duplicate Check)

触发条件：c2e_addr_payload_dupchek_valid = 1 (来自 atc_ctrl 的 insert 路径)

流程 (遍历所有 32 sets × 64 ways = 2048 entries)：
  1. 提取 translation completion 中的 {PV, PASID, device_function_ID, STU}
  2. 对未转换地址按 STU 进行掩码处理
  3. 遍历全部 entry，按命中规则进行匹配
  4. 任一 entry 匹配 → e2c_addr_duplicate = 1, e2c_dup_entry_idx = 命中索引
  5. 全部不匹配   → e2c_addr_duplicate = 0

⚠ 1GHz 时序注意：
  全遍历 2048 entry 比较在单周期不可行。采用 pipeline 分 4 拍完成，
  每拍比较 8 个 set (512 entry)。dupcheck 总延迟 = 4 cycles。

结果处理：
  ┌─ 地址不重复 → atc_ctrl NRU 分配新 entry，c2e_update_entry 写入
  └─ 地址重复   → atc_ctrl 通过 c2e_update_entry 覆盖该 entry (更新 STU/perm)

5.4 Entry 更新 (Insert)

当 c2e_update_entry 有效时，将目标 entry 更新为以下内容。同时， NRU 状态更新为 ACTIVE ({1,1}) 【v2.4】， 防止同一 entry 在下次 NRU 选择时被再次选为 victim（避免连续 insert 覆盖同一 way）。

字段位宽来源
PV1bATS Completion → PV 字段
PASID20bATS Completion → PASID 字段
device_function_ID16bATS Completion → Function ID (BDF)
untranslated_addr64bATS Completion → 未转换地址 (掩码处理前原值)
STU5bATS Completion → STU (编码: STU = log2(page_size))
translated_addr64bATS Completion → 已转换 HPA

5.5 Entry 失效 (Invalidation)

5.5.1 常规失效 (ATS Invalidation Request)

触发条件：RC 发来 ATS Invalidation Request，经 atc_ctrl 分发

失效命中判断逻辑 (与查表规则对称)：

  CASE 1: 失效请求中 PV 无效
    ├── 表中 PV 有效 → 仅比较 device_function_ID（通过则失效）
    └── 表中 PV 无效 → 比较 device_function_ID AND untranslated_addr（通过则失效）

  CASE 2: 失效请求中 PV 有效
    └── 比较 PV AND PASID AND device_function_ID AND untranslated_addr
        （全部通过则失效）

执行：
  ├── 清空命中 entry 的 valid 位和地址信息
  ├── 将失效命中信息反馈给 atc_ctrl (e2c_inv_valid, e2c_inv_hit_idx)
  ├── 失效完成后清除 NRU 状态 (used=0, not_last=1)
  └── 失效完成后可向该 entry 存入新的 translation

5.5.2 ATS Enable 切换失效

触发条件：ats_enable_toggle = 1 (ats_enable 发生 0→1 或 1→0 变化)

行为：
  ┌─ ats_enable 0→1 (ATS 刚使能)：清除所有 entry (全局无效)，从干净状态开始
  └─ ats_enable 1→0 (ATS 禁用)：清除所有 entry，DMA 后续使用 1:1 直通 (identity map)

实现：
  ats_enable_toggle 有效 → 遍历全部 2048 entry → valid 清 0
  多 cycle 完成 (如 4 cycle 清理 512 entry/cycle)

5.5.3 FLR (Function Level Reset) 失效 【新增独立信号】

触发条件：flr_req = 1, flr_func_id = 目标 Function ID

行为：
  遍历全部 2048 entry，清除所有 device_function_ID 匹配的 entry：
    for each entry:
      if entry.valid AND entry.func_id == flr_func_id:
        entry.valid  <= 0
        entry.nru    <= {0, 1}   // 重置 NRU 状态

与 ATS toggle 的区别：
  ├── ATS toggle：清除 全部 entry (全局操作)
  └── FLR：仅清除 指定 Function 的 entry (选择性操作)

接口设计理由：
  FLR 通过独立信号直接触发而非间接通过 ATS disable 实现，原因：
  1. 语义清晰，FLR 是独立的 PCIe reset 事件
  2. 响应速度快，无需等待 ATS enable 边沿检测
  3. 可精确清除单个 Function 的 entry，不影响其他 Function

7.2 NRU (Not Recently Used) 替换策略

每个 entry 维护 2b NRU 状态 (编码: {used, not_last})：
  ┌────────┬──────────┬──────────┬──────────────────────────────┐
  │  used  │ not_last │  编码     │           含义                 │
  ├────────┼──────────┼──────────┼──────────────────────────────┤
  │   0    │    1     │  FREE    │ 空闲 (复位值) / 最高替换优先级  │  ← 优先替换
  │   0    │    0     │  IDLE    │ 衰减后冷条目 / 次高替换优先级   │
  │   1    │    1     │  ACTIVE  │ 活跃 (insert/lookup 命中置位)  │
  │   1    │    0     │  PROTECT │ 保护 (用于 pin 住关键 entry)   │
  └────────┴──────────┴──────────┴──────────────────────────────┘

替换选择流程 (victim search, 按优先级递减):
  1. FREE  (used=0, not_last=1) → 优先替换
  2. IDLE  (used=0, not_last=0) → 次优先替换
  3. ACTIVE(used=1, not_last=1) → 再次之
  4. 任意有效 entry (兜底)

访问更新:
  ├── Lookup 命中: 命中 way → ACTIVE ({1,1})
  ├── Insert 写入: 写入 way → ACTIVE ({1,1}) 【v2.4 新增】
  ├── nru_clear_all_used: 仅清除所有 used 位; 第 0 路额外清除 not_last (防活锁)
  └── 定期衰减 (NRU_DECAY_INTERVAL=1024): 【v2.4 修正】
      只衰减 nru_used==1 的条目 (ACTIVE/PROTECT → IDLE):
        nru_used[i]     <= 0
        nru_not_last[i] <= 0
      FREE 条目 (used=0, not_last=1) 和 IDLE 条目 (used=0, not_last=0) 保持不变,
      确保 FREE 优先级始终高于衰减后的 IDLE, 保证空闲 slot 被优先复用。

8. 时序与流水线设计 (SF4X @1GHz) 【新增章节】

1GHz 时序约束分析

SF4X (Samsung 4nm-class) 工艺，典型标准单元延迟 ~8-12ps/gate。1ns 周期约对应 80-120 级逻辑门深度。以下分析各路径的时序可行性。

v2.3 变更: Sync SRAM 分离

8.3 物理实现建议 【v2.3 更新】

  • TAG/数据分离存储：TAG (107b/entry) 使用寄存器阵列实现组合比较；DATA (68b/entry) 使用 Sync SRAM macro (2048×68b)。SRAM 读延迟 1 cycle，不占用 S1 关键路径
  • 比较器布局：64 way × 17 addr = 1088 个 64b 比较器围绕 TAG 寄存器阵列排列，物理上靠近寄存器输出端口，减少走线延迟
  • Hit 编码器：使用分层优先编码器 (8×8 + 8:1)，减少级联深度。17 组 hit vector 共享编码器结构
  • 流水线寄存器：S0/S1、S1/S2、S2/S3 之间插入寄存器。S1 寄存器捕获 TAG 比较结果；S2 寄存器捕获 SRAM 返回数据；S3 寄存器驱动输出
  • Clock gating：仅被选中的 set 进行 way 比较和 SRAM 读，其余 31 个 set 的 TAG 比较器和 SRAM clock-gated

12. 用户分区功能 【v0.3 新增】

设计目标

在 SR-IOV 多 VF 场景下，通过 csr_num_users (3-bit) 将 2048 个 entry 平均切分为独立分区。每个 VF 只能访问自己分区内的 entry，实现硬件级隔离，防止单 VF 驱逐其他 VF 的 cache entry。

12.2 切分策略

12.2.1 Set 级切分 (≤32 用户)

每个用户独占 32/N 个完整 set（含全部 64 way）。Hash 函数 partition_hash() 将请求映射到用户分区内的 set:

user_id      = func_id[5:0] % N
sets_per_user = 32 / N
base_set     = user_id * sets_per_user
local_set    = raw_hash % sets_per_user
final_set    = base_set + local_set

12.2.2 Set+Way 混合切分 (>32 用户)

每个 set 内按 way 切分。以 64 用户为例: user 0 占 ways 0-31, user 1 占 ways 32-63。48 用户复用相同硬件。

users_per_set = ceil(N/32)
ways_per_user = 64 / users_per_set
way_base      = (user_id % users_per_set) * ways_per_user
way_limit     = way_base + ways_per_user
// NRU victim 限定在 [way_base, way_limit)

12.3 分区感知模块

模块分区影响
atc_lookup_engineHash 使用 partition_hash()，自动映射到用户 set 范围
atc_ctrlInsert hash 使用分区映射；NRU way_base/limit 计算
atc_nru_replacerVictim 搜索限定 [way_base, way_limit)；条件屏蔽兼容 DC
atc_inv_handler失效扫描从 [0,31] 缩小到 [base, limit)；ATS toggle 全局
atc_csr_ifcsr_num_users pass-through 到 cfg_num_users
重要约束
  • 动态切换: csr_num_users 变更后建议通过 ATS toggle 清除全部 entry 后再切换分区模式
  • 48 用户限制: 复用 64 用户硬件，48×32=1536 条目，512 条目未使用
  • NRU 兼容: 分区搜索使用条件屏蔽 (in_range) 而非变量 bounds，满足 DC 综合 ELAB-800 约束
